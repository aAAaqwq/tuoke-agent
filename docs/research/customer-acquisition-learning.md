## 第7轮：触达自动化与多渠道整合

**学习时间**：2026-03-09 10:04 (Asia/Shanghai)

### 1. 多渠道触达策略

#### 电话触达
- **智能外呼系统**
  - AI语音外呼：TTS + ASR + NLP 组合
  - 预测式拨号：根据接通率自动调节拨打频率
  - 并发外呼：支持多线路同时拨打
  - 智能路由：根据客户画像分配最优话术

- **通话记录与分析**
  - 全量录音存储（合规前提下）
  - 语音转文字（ASR）
  - 情绪识别：检测客户态度变化
  - 关键词提取：需求、异议、意向信号
  - 通话质量评分：话术执行度评估

- **技术栈参考**
  - Twilio / 阿里云语音 / 腾讯云语音
  - SIP中继、WebRTC
  - VAD（语音活动检测）、降噪

#### 邮件触达
- **批量邮件发送**
  - 模板系统：变量替换（{{name}}, {{company}}）
  - 发送节奏控制：避免触发垃圾邮件过滤
  - 域名预热：新域名逐步提升发送量
  - SPF/DKIM/DMARC 配置

- **追踪分析**
  - 打开率（Open Rate）：追踪像素
  - 点击率（Click Rate）：链接追踪
  - 回复率：最重要的质量指标
  - 退信率（Bounce Rate）：邮箱质量监控
  - 取消订阅率：内容相关性反馈

- **工具参考**
  - SendGrid / Mailgun / Amazon SES
  - Hunter / Apollo.io（邮箱验证）
  - Lemlist / Mailshake（冷邮件自动化）

#### 社交触达
- **微信**
  - 企业微信：合规的 B2B 触达渠道
  - 自动欢迎语、关键词回复
  - 朋友圈定时发布
  - 限制：每日添加好友数、消息频率
  - 风控：避免批量操作触发封号

- **LinkedIn**
  - InMail：付费触达非联系人
  - Connection Request：好友申请 + 个性化备注
  - 每周邀请限制（约 100 个）
  - 内容互动：点赞评论建立信任
  - Sales Navigator：高级搜索与线索管理

- **其他平台**
  - Twitter/X：公开互动、DM
  - Discord/Telegram：社区运营
  - 抖音/小红书：私信限制较严

#### 短信触达
- **合规性要求**
  - 用户同意：事前授权或合理预期
  - 退订机制：必须包含"回T退订"
  - 时间限制：避免夜间发送（通常 8:00-21:00）
  - 内容审核：敏感词过滤

- **批量发送**
  - 短链追踪：点击行为分析
  - 分批发送：避免通道拥堵
  - 失败重试：运营商通道问题
  - 号码清洗：过滤无效号码

- **服务商参考**
  - 阿里云短信 / 腾讯云短信
  - Twilio / Nexmo（国际）
  - 变量模板、签名审核

### 2. 触达自动化

#### 触达时机优化
- **最佳发送时间**
  - B2B邮件：周二至周四上午 9-11 点
  - B2C短信：午休（12-13点）、晚间（18-20点）
  - 电话：上午 10-11 点、下午 14-16 点
  - 地区时区：按客户所在地调整

- **行为触发**
  - 网站访问后 24 小时内
  - 内容下载后即时跟进
  - 试用期即将结束
  - 长时间未活跃（挽回）

- **频率控制**
  - 单渠道频率上限（如邮件每周不超过 2 封）
  - 跨渠道去重：避免同一天多渠道轰炸
  - 冷却期：拒绝后暂停触达 30 天

#### 内容个性化
- **基础变量**
  - 姓名、公司、职位
  - 行业、地区、规模

- **行为变量**
  - 最近访问页面
  - 下载过的内容
  - 参加过的活动

- **智能个性化**
  - AI 生成定制开场白
  - 基于领英信息的共同点挖掘
  - 公司新闻/动态融入邮件

- **示例模板**
  ```
  Hi {{first_name}}，

  看到 {{company}} 最近在 {{news_topic}} 方面有动作，
  我们的解决方案曾帮助 {{similar_company}} 实现 {{result}}，
  不知道是否值得聊聊？
  ```

#### A/B 测试
- **测试要素**
  - 邮件标题：直接影响打开率
  - 发送时间：不同时段对比
  - 行动号召（CTA）：按钮文案、位置
  - 内容长度：短 vs 长
  - 个性化程度：模板 vs 定制

- **测试方法**
  - 样本量：每组至少 500 个样本
  - 单变量测试：每次只改一个因素
  - 统计显著性：p < 0.05
  - 持续迭代：胜出者成为新基准

- **工具**
  - Mailchimp A/B Testing
  - Optimizely
  - 自建：Python + 统计库

#### 效果追踪
- **核心指标**
  - 触达率：成功送达/总发送
  - 打开率：打开/送达
  - 响应率：回复/打开
  - 转化率：成交/触达

- **漏斗分析**
  ```
  触达 → 打开 → 点击 → 回复 → 预约 → 成交
  每个环节识别流失点
  ```

- **归因模型**
  - 首次触达归因
  - 末次触达归因
  - 多触达线性归因
  - 时间衰减归因

- **报表仪表盘**
  - 渠道对比
  - 趋势变化
  - 异常预警（如打开率骤降）

### 3. 响应处理

#### 自动回复机制
- **即时确认**
  - "收到您的咨询，我们将在 24 小时内回复"
  - 工作时间外自动应答

- **智能路由**
  - 意向客户 → 销售主管
  - 技术问题 → 技术支持
  - 投诉 → 客服团队

- **FAQ 自动应答**
  - 关键词匹配
  - 意图识别（NLU）
  - 知识库检索

#### 意向识别
- **意向信号**
  - 正向：询问价格、要求演示、索要案例
  - 中性：询问功能、对比竞品
  - 负向：明确拒绝、投诉

- **NLP 分类**
  - 训练数据：历史对话标注
  - 模型：BERT / GPT 分类头
  - 输出：意向等级 + 置信度

- **规则引擎**
  - 关键词规则：价格词 → 高意向
  - 句式规则：疑问句 + 产品词
  - 组合规则：多个弱信号 → 强信号

#### 线索分级
- **BANT 框架**
  - Budget（预算）
  - Authority（决策权）
  - Need（需求）
  - Timeline（时间线）

- **打分模型**
  - 显性属性：公司规模、职位
  - 行为分数：网站访问、内容下载
  - 互动分数：邮件回复、电话接通
  - 总分 → A/B/C/D 等级

- **动态调整**
  - 根据后续行为升降级
  - 长期无互动自动降级
  - 成交后转为客户标签

#### 人工介入时机
- **高意向即时通知**
  - A级线索 → 立即通知销售
  - 请求演示 → 自动创建日历邀请

- **复杂问题转人工**
  - AI 置信度 < 阈值
  - 涉及价格谈判
  - 投诉类消息

- **协作模式**
  - AI 草拟回复 → 人工审核发送
  - 人工接管对话 → AI 退居辅助
  - 对话摘要：关键信息提炼给人工

### 4. 合规与道德

#### 数据隐私保护
- **法规框架**
  - GDPR（欧盟）：最严格
  - CCPA（加州）：消费者数据权利
  - PIPL（中国）：个人信息保护法

- **核心原则**
  - 最小必要：只收集必要数据
  - 目的限制：不得超范围使用
  - 存储期限：到期即删除
  - 安全措施：加密、访问控制

- **实践要点**
  - 隐私政策：清晰告知
  - 数据地图：知道数据在哪
  - 定期审计：检查合规性

#### 反骚扰规范
- **中国法规**
  - 《通信短信息服务管理规定》
  - 未经同意不得发送商业短信
  - 必须提供退订方式

- **国际规范**
  - CAN-SPAM（美国）：商业邮件规范
  - CASL（加拿大）：需明确同意
  - GDPR：用户有被遗忘权

- **内部规范**
  - 触达频率上限
  - 拒绝名单（DNC List）
  - 渠道偏好尊重

#### 用户同意机制
- **同意类型**
  - 明示同意：主动勾选、签字
  - 默示同意：基于已有业务关系
  - 退出机制：opt-out 为主（部分场景需 opt-in）

- **同意记录**
  - 时间戳、来源、范围
  - 可追溯性：应对投诉和审计

- **同意范围**
  - 单用途 vs 多用途
  - 可转让性：能否共享给第三方

#### 数据删除权
- **被遗忘权（GDPR）**
  - 用户可要求删除其所有数据
  - 例外：法律留存、合法业务需要

- **中国 PIPL**
  - 个人有权请求删除
  - 处理者应当删除或匿名化

- **技术实现**
  - 软删除 vs 硬删除
  - 备份数据处理
  - 日志脱敏
  - 关联数据清理

## 第8轮：AI Agent 在拓客中的应用

**学习时间**：2026-03-09 10:06 (Asia/Shanghai)

### 1. AI Agent 拓客场景

#### 智能客服：初步沟通与需求了解
- **7×24 小时在线接待**
  - 无需人工值守，随时响应客户咨询
  - 多语言支持：自动检测并切换语言
  - 高并发处理：同时接待数百个对话

- **需求挖掘对话流**
  ```
  客户：你们有什么产品？
  Agent：我们提供 [产品类别]，请问您主要想解决什么问题？
  客户：我们销售团队效率低
  Agent：了解，是线索获取、跟进还是转化环节的问题？
  → 逐步定位痛点
  ```

- **上下文记忆**
  - 会话内记忆：记住对话历史
  - 跨会话记忆：识别回头客，延续上次话题
  - 用户画像积累：每次互动丰富客户档案

- **人工无缝交接**
  - 复杂问题自动转人工
  - 对话摘要同步给客服
  - 客服接管后 Agent 退居辅助

#### 意向识别：对话分析与情绪识别
- **实时意向打分**
  - 正向信号：询价、要求演示、索要案例、问实施周期
  - 负向信号：只问免费功能、语气不耐烦、明确拒绝
  - 中性信号：一般功能咨询、竞品对比

- **情绪感知**
  - 语音情绪识别（电话场景）：语调、语速、停顿
  - 文本情绪分析：感叹词、标点、用词倾向
  - 情绪变化追踪：对话过程中态度转变

- **意图分类模型**
  ```
  输入：客户消息 + 历史对话
  输出：{
    intent: "询价" | "功能咨询" | "投诉" | "闲聊",
    sentiment: "positive" | "neutral" | "negative",
    urgency: "high" | "medium" | "low",
    confidence: 0.85
  }
  ```

- **触发规则引擎**
  - 高意向 + 高紧急度 → 立即通知销售
  - 投诉 → 转客服主管 + 预警
  - 闲聊 → 继续自动化引导

#### 信息收集：企业调研与联系人挖掘
- **企业背景自动调研**
  - 工商信息：天眼查、企查查 API
  - 官网爬取：业务介绍、产品、团队
  - 新闻动态：最近融资、扩张、人事变动
  - 社交媒体：公众号、微博、LinkedIn 公司页

- **关键人识别**
  - 决策者：CEO、VP、总监级
  - 影响者：技术负责人、采购负责人
  - 使用者：一线员工、部门主管

- **联系信息验证**
  - 邮箱格式验证 + SMTP 探测
  - 电话有效性检测
  - 社交账号活跃度

- **调研报告自动生成**
  ```markdown
  ## XX科技有限公司 调研报告

  ### 基本信息
  - 成立：2018年，B轮，500人
  - 业务：企业 SaaS，主营 XX 解决方案

  ### 最近动态
  - 2026.02 完成B轮融资 5000万
  - 2026.01 新任销售总监上任

  ### 关键联系人
  - 张XX，销售总监，原 XX 公司
  - 李XX，技术VP，可尝试技术切入

  ### 推荐策略
  - 时机：刚融资，扩张期，预算充足
  - 切入点：新销售总监可能正在评估工具
  ```

#### 跟进提醒：时机优化与话术建议
- **智能跟进时机**
  - 基于历史数据学习最佳间隔
  - 行为触发：客户打开邮件、访问官网
  - 日历事件：客户公司财报日、行业展会
  - 冷却规则：拒绝后暂停 N 天

- **动态话术生成**
  ```
  背景：客户 3 天前打开了我的方案邮件但未回复

  AI 建议：
  "王总，看到您查看了方案，想确认下是否有需要补充的信息？
   另外我们最近帮 [同行业客户] 实现了 XX 效果，
   如果感兴趣可以安排 15 分钟演示。"
  ```

- **多渠道协同提醒**
  - 电话未接通 → 建议发短信
  - 邮件 3 次未打开 → 建议换 LinkedIn
  - 长期无响应 → 标记为低优先级

- **话术效果追踪**
  - 哪些话术获得回复
  - 哪些话术导致拉黑
  - A/B 测试不同开场白

### 2. LLM 在拓客中的能力

#### 企业信息理解与总结
- **非结构化信息处理**
  - 官网文本提取关键信息
  - 新闻报道情感和事件提取
  - 财报数据解读

- **智能摘要**
  ```
  输入：某公司 10 篇新闻报道（5000字）
  输出：
  - 2025年营收增长 40%，主要来自海外市场
  - 正在拓展东南亚业务，计划新加坡设立分部
  - CTO 离职，技术团队在招聘
  ```

- **关联推理**
  - 公司最近融资 → 可能有采购预算
  - 招聘销售团队 → 可能在评估销售工具
  - 参加 XX 行业大会 → 可安排会面

- **竞品对比分析**
  - 自动识别客户正在使用的竞品
  - 生成差异化优势说明
  - 针对性话术建议

#### 个性化内容生成
- **冷邮件生成**
  ```
  输入：目标公司信息 + 我们产品卖点
  输出：
  "李总您好，

  看到 [公司名] 最近在 [动态]，想必 [痛点假设]。
  我们的 [产品] 曾帮助 [类似公司] 实现 [具体结果]，
  不知道是否值得 15 分钟聊聊？

  祝好，
  Daniel"
  ```

- **LinkedIn 连接申请**
  - 基于共同点生成个性化备注
  - 提及：共同好友、校友、行业活动

- **跟进内容变体**
  - 同一意图生成多个版本
  - 语气调节：正式/轻松/直接
  - 长短版本：短信 vs 邮件

- **内容本地化**
  - 翻译 + 文化适配
  - 地区特定案例引用

#### 对话式销售助手
- **销售实时辅助**
  - 通话中实时提示关键信息
  - 客户提到竞品 → 自动弹出对比资料
  - 客户提异议 → 推荐回应话术

- **会后纪要生成**
  ```
  会议时间：2026-03-09 14:00
  参与方：我方-Daniel，客户-王总、李经理

  关键讨论：
  1. 客户当前痛点：销售线索质量差，转化率仅 5%
  2. 决策时间线：Q2 之前需要敲定方案
  3. 预算范围：年费 20-30 万可接受

  下一步：
  - [ ] 发送详细方案（3/10 前完成）
  - [ ] 安排产品演示（3/15）
  - [ ] 对接技术团队做 POC
  ```

- **谈判策略建议**
  - 客户压价 → 提供价值锚点
  - 竞品介入 → 强调差异化
  - 决策拖延 → 制造紧迫感

#### 转化率漏斗

**标准拓客漏斗**
```
曝光（Impression）
   │ 转化率：2-5%
   ↓
点击/访问（Traffic）
   │ 转化率：10-30%
   ↓
线索获取（Lead）
   │ 转化率：20-40%
   ↓
MQL（市场确认线索）
   │ 转化率：30-50%
   ↓
SQL（销售确认线索）
   │ 转化率：20-40%
   ↓
商机/机会（Opportunity）
   │ 转化率：20-40%
   ↓
成交（Customer）

整体转化率 = 各环节转化率相乘
例：5% × 20% × 30% × 40% × 30% × 30% = 0.01%
即 10,000 次曝光 → 1 个客户
```

**各环节优化重点**
| 环节 | 优化方向 | 关键指标 |
|-----|---------|---------|
| 曝光→点击 | 广告创意、标题优化 | CTR |
| 点击→线索 | 落地页设计、表单简化 | 转化率 |
| 线索→MQL | 自动化评分、快速响应 | 响应时间 |
| MQL→SQL | 销售跟进、需求确认 | 跟进率 |
| SQL→商机 | 方案演示、价值传递 | 推进率 |
| 商机→成交 | 价格谈判、消除异议 | 成交率 |

**漏斗异常诊断**
```
问题：线索数量多但 SQL 少
可能原因：
- 线索来源质量差
- MQL 评分标准过松
- 销售跟进不及时
- 线索与产品不匹配

问题：SQL 多但成交少
可能原因：
- 产品竞争力不足
- 定价策略问题
- 销售能力待提升
- 竞品介入
```

#### 获客成本（CAC）

**CAC 计算公式**
```
CAC = 总营销销售成本 / 新客户数

成本包括：
- 广告投放费用
- 营销工具/平台费用
- 销售团队薪资（含提成）
- 内容制作成本
- 活动费用
- 技术基础设施成本

时间范围：
- 通常按月/季度计算
- 注意成本与客户归因的时间匹配
```

**分渠道 CAC**
```
渠道          投入      新客户    CAC
---------------------------------------
百度SEM       50,000    25       2,000
微信广告      30,000    18       1,667
内容营销      20,000    15       1,333
线下活动      40,000    12       3,333
转介绍        5,000     20         250
---------------------------------------
合计         145,000    90       1,611
```

**CAC 优化策略**
1. **降低成本**
   - 优化广告投放精准度
   - 提升内容复用率
   - 自动化替代人工
   - 淘汰低效渠道

2. **提升转化**
   - 缩短销售周期
   - 提高各环节转化率
   - 减少线索流失

3. **关注 LTV/CAC 比值**
   - 健康比值：LTV / CAC ≥ 3
   - 回本周期：CAC / 月度利润 < 12 个月

**CAC 常见误区**
- ❌ 只算广告费，忽略人工成本
- ❌ 用总客户数，混入自然流量
- ❌ 时间窗口不匹配（广告当月，成交跨月）
- ❌ 忽略客户质量（低 CAC 可能是低质量客户）

#### 客户生命周期价值（LTV）

**LTV 计算方法**
```
简单公式：
LTV = ARPU × 客户生命周期（月）

ARPU = 总收入 / 活跃客户数

详细公式：
LTV = (ARPU × 毛利率) / 流失率

例：
- 月 ARPU：10,000 元
- 毛利率：70%
- 月流失率：2%

LTV = (10,000 × 0.7) / 0.02 = 350,000 元
```

**LTV 分层分析**
```
客户分层         LTV      占比    贡献收入
----------------------------------------
企业客户      500,000    10%      50%
中型客户      150,000    30%      35%
小型客户       50,000    60%      15%

洞察：
- 10% 客户贡献 50% 收入
- 应重点投入高价值客户获取
```

**提升 LTV 策略**
1. **提升客单价**
   - 增购/交叉销售
   - 版本升级引导
   - 长期合约折扣

2. **延长生命周期**
   - 客户成功团队
   - 定期回访
   - 增值服务

3. **降低流失率**
   - 流失预警
   - 挽回机制
   - 产品改进

**LTV/CAC 比值解读**
```
LTV/CAC < 1  ：商业模式不可持续
LTV/CAC = 1-2：勉强生存，优化空间大
LTV/CAC = 3-5：健康增长
LTV/CAC > 5  ：可能投入不足，错失增长机会
```

**漏斗分析维度**

**漏斗瓶颈识别**
```
方法 1：与行业基准对比
- 访问→线索：行业 15%，我方 10% → 优化落地页
- SQL→商机：行业 40%，我方 50% → 表现良好

方法 2：同比环比分析
- 本月访问→线索从 15% 跌到 10%
- 可能原因：页面改版、流量质量下降

方法 3：细分分析
- 整体转化率 3%
- A渠道 5%，B渠道 1%
- 应增加 A 渠道投入
```

**漏斗预测模型**
```python
# 简单预测：基于历史转化率
def predict_revenue(current_leads, historical_rates):
    """
    current_leads: 当前漏斗各层数量
    historical_rates: 历史各环节转化率
    """
    prediction = current_leads
    for rate in historical_rates:
        prediction = prediction * rate
    return prediction

# 例：当前 1000 线索，预计成交 15 个
# 1000 × 25% × 40% × 35% × 30% = 10.5
```

#### A/B 测试方法论

**A/B 测试流程**
```
1. 提出假设
   "邮件标题加数字能提升打开率"

2. 设计实验
   - 对照组：原标题
   - 实验组：标题加数字
   - 样本量：每组 1000

3. 执行测试
   - 随机分组
   - 同时发送（避免时间偏差）
   - 收集数据

4. 统计分析
   - 计算显著性（p-value）
   - 确认样本量足够

5. 得出结论
   - 差异显著 → 采用胜出方案
   - 差异不显著 → 继续测试或放弃
```

**统计显著性判断**
```
概念：
- 置信度：差异不是随机产生的概率
- 通常要求 95% 置信度（p < 0.05）

示例：
对照组：1000 人，打开 150 人（15%）
实验组：1000 人，打开 180 人（18%）

提升：3 个百分点（相对提升 20%）
显著性检验：p = 0.03 < 0.05
结论：差异显著，采用实验组方案
```

**样本量计算**
```
最小样本量公式：
n = (Zα/2 + Zβ)² × 2 × p × (1-p) / δ²

简化估算表（95%置信度，80%功效）：

基础转化率    期望提升    每组最小样本
--------------------------------------
5%           20%        15,000
5%           50%         2,400
10%          20%         7,200
10%          50%         1,200
20%          20%         3,000
20%          50%           500

关键洞察：
- 基础转化率越低，需要样本越大
- 期望提升越小，需要样本越大
- 小提升需要大量数据才能验证
```

**常见测试场景**
| 测试对象 | 变量 | 指标 |
|---------|-----|------|
| 落地页 | 标题、CTA按钮、表单长度 | 转化率 |
| 邮件 | 标题、发件人、发送时间 | 打开率、点击率 |
| 广告 | 创意、文案、受众定向 | CTR、CPC |
| 电话 | 话术开场白、时间段 | 接通率、预约率 |
| 定价 | 价格锚点、折扣方式 | 成交率、客单价 |

**A/B 测试陷阱**
1. **过早下结论**
   - 样本量不足就宣布胜者
   - 解决：预设样本量，达到后再分析

2. **多重比较问题**
   - 同时测试多个变量
   - 解决：多变量测试或 Bonferroni 校正

3. **新奇效应**
   - 短期提升来自新鲜感，非真实改进
   - 解决：延长观察期

4. **选择性报告**
   - 只报告成功测试
   - 解决：记录所有测试，包括失败的

#### 归因分析

**归因模型类型**
```
1. 末次触达归因（Last Touch）
   成功归因于成交前最后一个触点

   优点：简单明确
   缺点：忽视前期培育

   适用：短周期、低客单价

2. 首次触达归因（First Touch）
   成功归因于第一个触点

   优点：识别获客来源
   缺点：忽视后续影响

   适用：品牌营销评估

3. 线性归因
   所有触点平均分配功劳

   优点：公平
   缺点：忽视触点重要性差异

   适用：初步分析

4. 时间衰减归因
   越接近成交的触点权重越高

   优点：符合决策过程
   缺点：可能低估早期影响

   适用：长销售周期

5. 位置归因（U型）
   首次和末次权重高，中间平均

   优点：平衡获客和成交
   缺点：中间触点被低估

   适用：B2B 常见场景
```

**多触达归因示例**
```
客户旅程：
Day 1：点击百度广告
Day 7：打开邮件
Day 14：参加 Webinar
Day 21：销售电话
Day 28：成交

归因分配：
- 末次触达：销售电话 100%
- 首次触达：百度广告 100%
- 线性归因：各 25%
- 时间衰减：百度 10%，邮件 15%，Webinar 25%，电话 50%
- 位置归因：百度 40%，邮件 6.7%，Webinar 6.7%，电话 40%
```

**归因分析实践**
```python
# 简化归因计算
def attribution_analysis(customer_journeys, model='last_touch'):
    """
    customer_journeys: [
        {'customer_id': 1, 'touchpoints': [
            {'channel': 'baidu', 'day': 1},
            {'channel': 'email', 'day': 7},
            {'channel': 'sales', 'day': 21}
        ], 'revenue': 10000},
        ...
    ]
    """
    channel_credit = {}

    for journey in customer_journeys:
        touchpoints = journey['touchpoints']
        revenue = journey['revenue']

        if model == 'last_touch':
            # 末次触达
            last_channel = touchpoints[-1]['channel']
            channel_credit[last_channel] = channel_credit.get(last_channel, 0) + revenue

        elif model == 'linear':
            # 线性归因
            credit_per_touch = revenue / len(touchpoints)
            for tp in touchpoints:
                channel = tp['channel']
                channel_credit[channel] = channel_credit.get(channel, 0) + credit_per_touch

    return channel_credit
```

**归因分析注意事项**
- 建立统一客户 ID（跨设备/渠道）
- 定义触点分类标准
- 选择与业务匹配的模型
- 定期验证归因准确性

**渠道预算分配**
```

- 计算各渠道边际成本
- 在边际成本相等的点分配预算
```

**实验优先级评估**
```

- Impact（影响）：成功后收益多大
- Confidence（信心）：假设可靠程度
- Ease（难度）：实施难度

- Impact: 9（可能提升 30% 收入）
- Confidence: 4（不确定性大）
- Ease: 3（需2周+多方协调）
- 优先级 = 9×4×3 = 108

→ 优先做实验A
```

```python
    def decorator(func):
        last_called = [0.0]
        @wraps(func)
        def wrapper(*args, **kwargs):
```

```python
def dedupe_simple(records, key_field):
    seen = set()
    result = []
    for record in records:
        key = record.get(key_field)
        if key and key not in seen:
            seen.add(key)
            result.append(record)
    return result

# 模糊匹配去重（企业名称）
from difflib import SequenceMatcher

def similar(a, b, threshold=0.85):
    return SequenceMatcher(None, a, b).ratio() > threshold

def dedupe_fuzzy(companies):
    result = []
    for company in companies:
        is_dup = False
        for existing in result:
            if similar(company['name'], existing['name']):
                is_dup = True
                break
        if not is_dup:
            result.append(company)
    return result

# 合并重复记录
def merge_records(records):
    merged = {}
    for record in records:
        key = record['id']
        if key in merged:
            # 合并字段，保留最新/最完整的
            for k, v in record.items():
                if v and (k not in merged[key] or not merged[key][k]):
                    merged[key][k] = v
        else:
            merged[key] = record
    return list(merged.values())
```

```python
import re

def normalize_company_name(name):
    # 去除常见后缀
    suffixes = [
        r'有限公司$', r'股份有限公司$', r'有限责任公司$',
        r'集团$', r'公司$', r'Co\.?Ltd\.?$', r'Inc\.?$'
    ]
    for suffix in suffixes:
        name = re.sub(suffix, '', name, flags=re.IGNORECASE)

    # 去除空格和特殊字符
    name = re.sub(r'\s+', '', name)

    # 统一大小写（中文保持原样）
    if name.isascii():
        name = name.title()

    return name.strip()

# 手机号标准化
def normalize_phone(phone):
    # 去除非数字字符
    digits = re.sub(r'\D', '', phone)

    # 去除中国区号
    if digits.startswith('86') and len(digits) == 13:
        digits = digits[2:]

    # 验证格式
    if len(digits) == 11 and digits.startswith('1'):
        return digits
    return None  # 无效号码

def normalize_email(email):

    email = email.strip().lower()

        return email
    return None
```

```python
# 基于 Redis 的分布式 URL 队列
import redis
from urllib.parse import urlparse

class DistributedQueue:
    def __init__(self, redis_url='redis://localhost:6379'):
        self.redis = redis.from_url(redis_url)
        self.queue_key = 'crawl:queue'
        self.seen_key = 'crawl:seen'

    def add_url(self, url, priority=0):
        """添加 URL 到队列"""
        url_hash = hashlib.md5(url.encode()).hexdigest()

        # 检查是否已处理
        if self.redis.sismember(self.seen_key, url_hash):
            return False

        # 添加到队列
        self.redis.zadd(self.queue_key, {url: priority})
        return True

    def get_url(self):
        """获取下一个 URL"""
        result = self.redis.zpopmax(self.queue_key)
        if result:
            return result[0][0].decode()
        return None

    def mark_done(self, url):
        """标记 URL 已处理"""
        url_hash = hashlib.md5(url.encode()).hexdigest()
        self.redis.sadd(self.seen_key, url_hash)
```

**代理池管理**
```python
class ProxyPool:
    def __init__(self):
        self.proxies = []
        self.proxy_stats = {}  # 代理成功率统计

    def add_proxy(self, proxy):
        self.proxies.append(proxy)
        self.proxy_stats[proxy] = {'success': 0, 'fail': 0}

    def get_proxy(self):
        """根据成功率选择代理"""
        if not self.proxies:
            return None

        # 按成功率排序
        sorted_proxies = sorted(
            self.proxies,
            key=lambda p: self._success_rate(p),
            reverse=True
        )
        return sorted_proxies[0]

    def _success_rate(self, proxy):
        stats = self.proxy_stats.get(proxy, {'success': 0, 'fail': 0})
        total = stats['success'] + stats['fail']
        if total == 0:
            return 0.5
        return stats['success'] / total

    def report_success(self, proxy):
        self.proxy_stats[proxy]['success'] += 1

    def report_fail(self, proxy):
        self.proxy_stats[proxy]['fail'] += 1
        # 失败率过高则移除
        if self._success_rate(proxy) < 0.3:
            self.remove_proxy(proxy)

    def remove_proxy(self, proxy):
        if proxy in self.proxies:
            self.proxies.remove(proxy)
```

**增量更新实现**
```python
class IncrementalUpdater:
    def __init__(self, db):
        self.db = db
        self.checkpoint_key = 'last_sync_timestamp'

    def sync(self, data_source):
        # 获取上次同步时间
        last_sync = self.db.get(self.checkpoint_key) or 0
        current_time = time.time()

        # 获取增量数据
        incremental_data = data_source.fetch_updated(since=last_sync)

        # 处理增量
        for record in incremental_data:
            existing = self.db.get(record['id'])

            if existing:
                # 更新
                changes = self._merge(existing, record)
                if changes:
                    self.db.update(record['id'], record)
                    self._log_change(record['id'], changes)
            else:
                # 新增
                self.db.insert(record)

        # 更新检查点
        self.db.set(self.checkpoint_key, current_time)

        return {
            'new': len([r for r in incremental_data if not self.db.get(r['id'])]),
            'updated': len([r for r in incremental_data if self.db.get(r['id'])])
        }

    def _merge(self, existing, new):
        """合并数据，返回变化的字段"""
        changes = {}
        for key, new_val in new.items():
            old_val = existing.get(key)
            if old_val != new_val and new_val is not None:
                changes[key] = {'old': old_val, 'new': new_val}
        return changes
```

**管道实现**
```python
from dataclasses import dataclass
from typing import Callable, List

@dataclass
class PipelineStep:
    name: str
    processor: Callable
    on_error: str = 'skip'  # skip, stop, retry

class DataPipeline:
    def __init__(self):
        self.steps: List[PipelineStep] = []
        self.metrics = {
            'processed': 0,
            'success': 0,
            'failed': 0,
            'errors': []
        }

    def add_step(self, name: str, processor: Callable, on_error='skip'):
        self.steps.append(PipelineStep(name, processor, on_error))
        return self

    async def execute(self, data):
        result = data

        for step in self.steps:
            try:
                result = await step.processor(result) if asyncio.iscoroutinefunction(step.processor) else step.processor(result)
                self.metrics['processed'] += 1

            except Exception as e:
                self.metrics['failed'] += 1
                self.metrics['errors'].append({
                    'step': step.name,
                    'error': str(e),
                    'data': result
                })

                if step.on_error == 'stop':
                    raise
                elif step.on_error == 'skip':
                    continue
                elif step.on_error == 'retry':
                    # 简单重试逻辑
                    for _ in range(3):
                        try:
                            result = step.processor(result)
                            break
                        except:
                            pass

        self.metrics['success'] += 1
        return result

# 使用示例
```

import asyncio

class StreamProcessor:
    def __init__(self, stream_name, consumer_group):
        self.stream_name = stream_name
        self.consumer_group = consumer_group
        self.redis = redis.Redis()

    async def process_stream(self, handler):
        """持续处理流数据"""
        while True:
            # 读取消息
            messages = self.redis.xreadgroup(
                self.consumer_group,
                'consumer-1',
                {self.stream_name: '>'},
                count=10,
                block=5000
            )

            for stream, msgs in messages:
                for msg_id, data in msgs:
                    try:
                        # 处理消息
                        await handler(data)
                        # 确认处理完成
                        self.redis.xack(self.stream_name, self.consumer_group, msg_id)
                    except Exception as e:
                        logging.error(f"处理失败: {e}")
                        # 消息会保持 pending 状态

            await asyncio.sleep(0.1)
